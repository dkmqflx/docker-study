## General Docker Commands

```shell
# 현재 설치된 Docker의 버전을 출력합니다.
docker --version

# Docker 전체 시스템 정보를 출력합니다.
# 실행 중인 컨테이너 수, 이미지 수, 스토리지 드라이버 종류 등을 확인할 수 있습니다.
docker info

# 사용 가능한 모든 Docker 명령어와 옵션 목록을 출력합니다.
docker help
```

## Image Management

> **이미지(Image)란?**
> 컨테이너를 만들기 위한 '설계도'입니다. 예를 들어 nginx 이미지를 받아서 여러 개의 nginx 컨테이너를 실행할 수 있습니다.

```shell
# 로컬(내 컴퓨터)에 저장된 이미지 목록을 출력합니다.
docker images

# Docker Hub(공개 이미지 저장소)에서 이미지를 다운로드합니다.
# 예: docker pull nginx
docker pull [image-name]

# 현재 디렉토리의 Dockerfile을 사용해 이미지를 빌드하고 이름을 붙입니다.
# -t: 이미지에 이름(태그)을 붙이는 옵션
# . : Dockerfile이 있는 현재 디렉토리를 의미
docker build -t [image-name] .

# 이미지에 다른 이름이나 버전 태그를 붙입니다.
# 예: docker tag my-app my-app:v1.0
docker tag [source-image] [target-image:tag]

# 이미지를 Docker Hub 같은 레지스트리에 업로드합니다.
docker push [image-name]

# 이미지를 삭제합니다. (실행 중인 컨테이너가 사용 중인 이미지는 삭제 불가)
docker rmi [image-id]
```

## Container Management

> **컨테이너(Container)란?**
> 이미지를 실행한 인스턴스입니다. 이미지가 '설계도'라면 컨테이너는 그 설계도로 만든 '실제 건물'입니다.

```shell
# 현재 실행 중인 컨테이너 목록을 출력합니다.
docker ps

# 중지된 컨테이너를 포함해 모든 컨테이너 목록을 출력합니다.
docker ps -a

# 이미지로부터 컨테이너를 생성하고 실행합니다.
docker run [options] [image-name]
    # 자주 쓰이는 옵션들:
    -d                              # 백그라운드(detached)에서 실행. 터미널을 점유하지 않음
    -p [host-port]:[container-port] # 호스트 포트와 컨테이너 포트를 연결(포트 포워딩)
                                    # 예: -p 8080:80 → 내 컴퓨터의 8080으로 접속하면 컨테이너의 80 포트로 연결
    --name [container-name]         # 컨테이너에 이름을 지정 (없으면 Docker가 랜덤 이름 생성)
    -v [host-path]:[container-path] # 볼륨(저장소)을 마운트 → 아래 Volume 섹션에서 자세히 설명

# 컨테이너를 삭제합니다. (먼저 중지한 후 삭제해야 함)
docker rm [container-id/name]

# 실행 중인 컨테이너를 강제로 삭제합니다. (-f = force)
docker rm -f [container-id/name]

# 실행 중인 컨테이너를 중지합니다.
docker stop [container-id/name]

# 중지된 컨테이너를 다시 시작합니다.
docker start [container-id/name]

# 컨테이너를 재시작합니다. (stop → start를 한 번에)
docker restart [container-id/name]
```

## Container Interaction

```shell
# 실행 중인 컨테이너 안에서 명령어를 실행합니다.
# -it: 인터랙티브 터미널 모드 (직접 입력을 주고받을 수 있게 함)
# 예: docker exec -it [container-id] /bin/bash → 컨테이너 내부의 bash 셸을 열어 직접 조작
docker exec [container-id/name] [command]

# 컨테이너의 로그를 출력합니다.
docker logs [options] [container-id/name]
    --follow    # 로그를 실시간으로 스트리밍 (Ctrl+C로 종료)

# 실행 중인 컨테이너의 표준 입출력에 내 터미널을 연결합니다.
# exec와의 차이: exec는 새 프로세스를 실행, attach는 기존 메인 프로세스에 붙음
docker attach [container-id/name]

# 컨테이너를 즉시 강제 종료합니다.
# SIGKILL: 프로세스를 즉시 죽이는 신호 (저장 없이 강제 종료)
# SIGTERM: 정상 종료를 요청하는 신호 (프로세스가 정리 후 종료)
docker kill [options] [container-id/name]
    --signal, -s    # 보낼 신호를 지정 (기본값: SIGKILL)
```

## Debugging Commands

```shell
# 컨테이너, 이미지, 네트워크 등의 상세 정보를 JSON 형식으로 출력합니다.
# IP 주소, 볼륨 마운트 경로, 환경 변수 등을 확인할 때 유용
docker inspect [object-id/name]

# 실행 중인 컨테이너들의 CPU, 메모리 등 리소스 사용량을 실시간으로 출력합니다.
docker stats [container-id/name]

# 컨테이너 안에서 실행 중인 프로세스 목록을 출력합니다.
docker top [container-id/name]

# Docker 데몬에서 발생하는 이벤트를 실시간으로 출력합니다.
# 컨테이너 시작/중지, 이미지 pull 등의 이벤트 확인 가능
docker events
```

## Volume Commands

> ### Docker Volume이란?
>
> 컨테이너는 기본적으로 **일회용**입니다. 컨테이너를 삭제하면 그 안의 데이터도 모두 사라집니다.
>
> **Volume**은 이 문제를 해결하기 위한 Docker의 데이터 저장 공간입니다.
> 컨테이너 외부(호스트 또는 Docker가 관리하는 공간)에 데이터를 저장하므로,
> 컨테이너가 삭제되어도 데이터가 유지됩니다.
>
> ```
> [컨테이너 A] ──┐
>                ├──▶ [Volume: my_volume] ──▶ 데이터 영구 보존
> [컨테이너 B] ──┘
> ```
>
> **사용 사례:** 데이터베이스 파일, 업로드된 파일, 설정 파일 등 영구 보존이 필요한 데이터

```shell
# 새로운 볼륨을 생성합니다.
# Docker가 내부적으로 저장 공간을 만들어 관리합니다.
docker volume create my_volume

# 현재 존재하는 볼륨 목록을 출력합니다.
docker volume ls

# 특정 볼륨의 상세 정보를 출력합니다.
# 볼륨이 실제로 저장된 호스트 경로(Mountpoint) 등을 확인할 수 있습니다.
docker volume inspect my_volume

# 볼륨을 삭제합니다. (사용 중인 볼륨은 삭제 불가)
docker volume rm my_volume

# 어떤 컨테이너에서도 사용되지 않는 볼륨을 모두 삭제합니다.
# 디스크 공간을 정리할 때 유용 (주의: 복구 불가)
docker volume prune
```

### 실습 1: Named Volume 생성 및 데이터 유지 확인

> 컨테이너를 삭제해도 볼륨에 저장한 데이터가 남아있는지 확인하는 실습입니다.

```shell
# 1단계: 'my_volume'이라는 이름의 볼륨을 생성합니다.
docker volume create my_volume

# 2단계: busybox 컨테이너를 실행하고 볼륨에 파일을 씁니다.
# --rm  : 컨테이너가 종료되면 자동으로 삭제 (볼륨은 삭제되지 않음)
# -it   : 인터랙티브 터미널 모드 (직접 명령어를 입력하기 위해 필요)
# -v my_volume:/data : 'my_volume' 볼륨을 컨테이너 내부의 /data 경로에 연결
# busybox : 매우 가벼운 리눅스 이미지 (실습용으로 자주 사용)
# sh    : 컨테이너 실행 후 sh(셸)를 시작
docker run --rm -it -v my_volume:/data busybox sh

    # --- 여기서부터는 busybox 컨테이너 내부 ---

    # /data 디렉토리에 hello.txt 파일을 생성합니다.
    # /data 는 위에서 my_volume을 마운트한 경로이므로, 이 파일은 볼륨에 저장됩니다.
    echo "Hello, Docker Volumes!" > /data/hello.txt

    # 컨테이너를 종료합니다. --rm 옵션으로 인해 컨테이너는 자동 삭제됩니다.
    # 하지만 my_volume은 Docker가 계속 관리하므로 데이터는 남아있습니다.
    exit

# 3단계: 새로운 컨테이너를 실행해서 볼륨의 데이터가 유지되는지 확인합니다.
# (이전 컨테이너는 삭제됐지만, 같은 my_volume을 마운트하면 데이터에 접근 가능)
docker run --rm -it -v my_volume:/data busybox sh

    # --- 새로운 컨테이너 내부 ---

    # 이전 컨테이너에서 저장한 파일을 읽습니다.
    # "Hello, Docker Volumes!" 가 출력되면 볼륨이 정상적으로 동작하는 것입니다.
    cat /data/hello.txt
```

### 실습 2: 두 컨테이너 간 볼륨 공유

> 두 컨테이너가 같은 볼륨을 공유하면 데이터를 서로 주고받을 수 있습니다.
> 예: 하나의 컨테이너가 로그를 쓰고, 다른 컨테이너가 그 로그를 읽는 구조

```shell
# 1단계: 첫 번째 컨테이너에서 'shared_volume'에 파일을 씁니다.
# (볼륨을 미리 생성하지 않아도, -v 옵션에서 자동으로 생성됩니다)
docker run --rm -it -v shared_volume:/shared busybox sh

    # --- 첫 번째 컨테이너 내부 ---

    # /shared 경로(= shared_volume)에 파일을 저장합니다.
    echo "Shared Data" > /shared/shared_file.txt

    # 컨테이너를 종료합니다.
    exit

# 2단계: 두 번째 컨테이너에서 같은 볼륨을 마운트해 파일을 읽습니다.
docker run --rm -it -v shared_volume:/shared busybox sh

    # --- 두 번째 컨테이너 내부 ---

    # 첫 번째 컨테이너에서 저장한 파일을 읽습니다.
    # "Shared Data"가 출력되면 볼륨 공유가 성공한 것입니다.
    cat /shared/shared_file.txt
```
