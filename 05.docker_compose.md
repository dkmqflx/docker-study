# Docker Compose 완전 정복

---

## 1. Docker Compose란?

> **"여러 컨테이너로 구성된 앱을 YAML 파일 하나로 정의하고, 명령어 하나로 실행하는 도구"**

---

### 왜 필요한가?

실제 서비스는 컨테이너 하나로 돌아가지 않습니다.

예를 들어 일반적인 웹 서비스라면:

```
웹 서버 (Nginx)
    +
앱 서버 (Node.js / Python)
    +
데이터베이스 (MySQL)
    +
캐시 (Redis)
```

이 4개를 `docker run`으로 하나씩 실행하면:

```bash
docker run -d -p 80:80 nginx
docker run -d -p 3000:3000 --env-file .env my-app
docker run -d -p 3306:3306 -e MYSQL_ROOT_PASSWORD=secret -v db_data:/var/lib/mysql mysql
docker run -d -p 6379:6379 redis
```

- 명령어가 길고 복잡합니다.
- 팀원과 환경을 공유하기 어렵습니다.
- 컨테이너 순서, 네트워크, 볼륨을 일일이 맞춰야 합니다.

**Docker Compose**를 쓰면 위 모든 설정을 `docker-compose.yml` 파일 하나에 담고, 명령어 하나로 해결합니다.

```bash
docker compose up
```

---

### 비유로 이해하기

Docker Compose는 **악보**와 같습니다.

- 각 컨테이너 = 악기 (바이올린, 첼로, 피아노...)
- Docker Compose = 악보 (각 악기가 언제, 어떻게 연주할지 정의)
- `docker compose up` = 지휘자가 지휘봉을 내려치는 순간

악보(Compose 파일) 하나만 있으면 어느 오케스트라(환경)에서든 **같은 곡(서비스)** 을 연주할 수 있습니다.

## 2. Docker Compose의 핵심 특징 5가지

### 1. Multi-Container Orchestration (멀티 컨테이너 오케스트레이션)

여러 컨테이너를 **하나의 애플리케이션으로** 묶어서 관리합니다.  
각 서비스의 의존 관계, 네트워크, 볼륨을 한 파일에서 정의합니다.

```
docker-compose.yml
├── web (Nginx)
├── app (Node.js)    → 서로 같은 네트워크에 자동 연결
├── db  (MySQL)
└── cache (Redis)
```

### 2. Declarative Configuration (선언형 설정)

"어떻게 실행할지"가 아닌 **"어떤 상태여야 하는지"** 를 YAML로 선언합니다.  
Docker Compose가 알아서 그 상태를 만들어 줍니다.

```yaml
# "이런 상태여야 한다"고 선언하면 Docker가 알아서 맞춰줍니다.
services:
  db:
    image: mysql:8.0
    ports:
      - "3306:3306"
```

### 3. Simplified Workflow (단순화된 워크플로우)

복잡한 환경 설정을 명령어 2개로 끝냅니다.

```bash
docker compose up -d    # 전체 서비스 시작
docker compose down     # 전체 서비스 종료 및 정리
```

### 4. Portability (이식성)

`docker-compose.yml` 파일 하나만 공유하면 팀 전원이 **동일한 환경**을 재현할 수 있습니다.  
개발, 스테이징, 프로덕션 환경 모두 같은 파일을 기반으로 실행합니다.

### 5. Service Independence (서비스 독립성)

전체를 재시작하지 않고 **특정 서비스만** 재시작할 수 있습니다.

```bash
docker compose restart app    # app 서비스만 재시작
docker compose stop db        # db 서비스만 중지
```

## 3. docker-compose.yml 파일 구조

```yaml
# docker-compose.yml 기본 구조
services: # 실행할 컨테이너(서비스)들을 정의
  서비스이름:
    image: # 사용할 이미지
    build: # Dockerfile로 빌드할 경우 경로 지정
    ports: # 포트 연결 [호스트]:[컨테이너]
    environment: # 환경변수
    volumes: # 볼륨 연결
    depends_on: # 의존 서비스 (먼저 실행되어야 할 서비스)
    networks: # 연결할 네트워크

networks: # 커스텀 네트워크 정의 (생략 시 기본 네트워크 자동 생성)

volumes: # Named Volume 정의
```

## 4. 실전 예시

### 기본 예시 — Node.js + MySQL

```yaml
services:
  app:
    build: . # 현재 폴더의 Dockerfile로 빌드
    ports:
      - "3000:3000" # 로컬 3000 → 컨테이너 3000
    environment:
      - DB_HOST=db # 서비스 이름(db)으로 DB에 접근
      - DB_PASSWORD=secret
    depends_on:
      - db # db가 먼저 실행된 뒤 app 실행

  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: myapp
    volumes:
      - db_data:/var/lib/mysql # DB 데이터 영속화

volumes:
  db_data: # Named Volume 선언
```

> **`depends_on`의 주의점:**  
> 컨테이너가 **시작**되는 순서를 보장하지만, 서비스가 **준비 완료**될 때까지 기다리지는 않습니다.  
> MySQL이 완전히 초기화되기 전에 앱이 연결을 시도할 수 있습니다.

---

### 실전 예시 — Nginx + Node.js + MySQL + Redis

```yaml
services:
  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf # 설정 파일 Bind Mount
    depends_on:
      - app

  app:
    build: .
    environment:
      - NODE_ENV=production
      - DB_HOST=db
      - REDIS_HOST=cache
    depends_on:
      - db
      - cache

  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: secret
      MYSQL_DATABASE: myapp
    volumes:
      - db_data:/var/lib/mysql

  cache:
    image: redis:7-alpine
    volumes:
      - redis_data:/data

volumes:
  db_data:
  redis_data:
```

## 4.5 Dockerfile과 Docker Compose의 관계

### 플로우 이해하기

Dockerfile과 Docker Compose의 관계는 다음 순서로 진행됩니다:

```
Dockerfile (빌드 스펙)
    ↓
build: . (docker-compose가 빌드 실행)
    ↓
이미지 생성
    ↓
컨테이너 실행
```

### 실제 예시

**Dockerfile:**

```dockerfile
# 빌드 스테이지: Node.js로 React 앱 빌드
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# 서빙 스테이지: 빌드 결과물만 Nginx로 서빙
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

**docker-compose.yml:**

```yaml
services:
  app:
    build: . # ← Dockerfile을 빌드해서 이미지 생성
```

### 정리

|                          |                                                                   |
| ------------------------ | ----------------------------------------------------------------- |
| **Dockerfile**           | 이미지를 어떻게 빌드할지 정의 + 기본 실행 명령어 (CMD) 지정       |
| **docker-compose build** | Dockerfile을 읽어서 이미지 생성                                   |
| **docker-compose up**    | 생성된 이미지를 컨테이너로 실행 (`command`로 CMD 오버라이드 가능) |

## 5. Docker Compose 주요 명령어

```bash
# 모든 서비스를 빌드하고 시작합니다.
# -d: 백그라운드 실행 (detached mode)
docker compose up
docker compose up -d

# 이미지를 강제로 다시 빌드하고 시작합니다.
# 소스코드나 Dockerfile이 바뀌었을 때 사용합니다.
docker compose up --build

# 모든 서비스를 중지하고 컨테이너·네트워크를 삭제합니다.
# 볼륨은 기본적으로 삭제되지 않습니다.
docker compose down

# 볼륨까지 함께 삭제합니다. (DB 데이터 초기화 시 사용)
docker compose down -v

# 실행 중인 서비스 목록과 상태를 확인합니다.
docker compose ps

# 특정 서비스의 로그를 확인합니다.
docker compose logs app
docker compose logs -f app      # 실시간 로그 스트리밍

# 특정 서비스만 재시작합니다.
docker compose restart app

# 특정 서비스 컨테이너 안으로 접속합니다.
docker compose exec app bash
docker compose exec db mysql -u root -p

# 서비스를 실행하지 않고 일회성 명령어만 실행합니다.
docker compose run app node migrate.js
```

## 6. 환경변수 관리 — `.env` 파일

민감한 정보(비밀번호, API 키)는 `docker-compose.yml`에 직접 쓰지 않고  
`.env` 파일로 분리합니다.

```bash
# .env 파일
MYSQL_ROOT_PASSWORD=mysecretpassword
MYSQL_DATABASE=myapp
APP_PORT=3000
```

```yaml
# docker-compose.yml에서 .env 파일의 변수를 사용
services:
  app:
    ports:
      - "${APP_PORT}:3000" # .env의 APP_PORT 변수 참조

  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD} # .env 변수 참조
      MYSQL_DATABASE: ${MYSQL_DATABASE}
```

> `.env` 파일은 반드시 `.gitignore`에 추가해서 저장소에 올라가지 않도록 하세요.

## 7. 컨테이너 간 통신

Docker Compose로 실행된 서비스들은 **자동으로 같은 네트워크**에 연결됩니다.  
IP 주소 대신 **서비스 이름**으로 서로 통신할 수 있습니다.

```yaml
services:
  app:
    environment:
      - DB_HOST=db # 'db'라는 서비스 이름이 곧 호스트명
      - REDIS_HOST=cache # 'cache'라는 서비스 이름이 곧 호스트명

  db:
    image: mysql:8.0

  cache:
    image: redis:7-alpine
```

```
app 컨테이너에서:
  mysql -h db        → db 컨테이너로 연결
  redis-cli -h cache → cache 컨테이너로 연결
```

## 8. 개발 vs 프로덕션 환경 분리

환경별로 다른 설정이 필요할 때 Compose 파일을 분리합니다.

```bash
# 기본 파일 (공통 설정)
docker-compose.yml

# 개발 환경 전용 (Bind Mount, 디버그 포트 등)
docker-compose.override.yml     # docker compose up 시 자동으로 적용됨

# 프로덕션 환경 전용
docker-compose.prod.yml
```

```bash
# 프로덕션 환경으로 실행
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

## 9. 핵심 정리

| 개념                      | 핵심                                            |
| ------------------------- | ----------------------------------------------- |
| **Docker Compose**        | 멀티 컨테이너 앱을 YAML로 정의하고 한 번에 실행 |
| **docker-compose.yml**    | 서비스·네트워크·볼륨을 선언하는 설정 파일       |
| **`docker compose up`**   | 전체 서비스 시작                                |
| **`docker compose down`** | 전체 서비스 종료 및 정리                        |
| **서비스 이름**           | 컨테이너 간 통신 시 호스트명으로 사용           |
| **`.env` 파일**           | 민감한 설정값을 파일로 분리                     |

> **핵심 한 문장:** Docker Compose는 여러 컨테이너의 복잡한 실행 명령을 YAML 파일로 코드화해서, 어디서든 `docker compose up` 하나로 동일한 환경을 재현하게 해줍니다.
