# Dockerfile 완전 정복

---

## 1. Dockerfile이란?

> **"Docker 이미지를 만들기 위한 설계도(레시피)"**

Dockerfile은 Docker 이미지를 빌드하는 방법을 단계별로 기술한 **텍스트 파일**입니다.  
파일 안의 각 명령어(instruction)는 이미지를 만드는 하나의 단계를 정의하며,  
이를 실행하면 항상 **동일한 이미지**가 만들어집니다.

### 비유로 이해하기

| 개념 | 비유 |
|---|---|
| Dockerfile | 요리 레시피 |
| `docker build` | 레시피대로 요리하는 행위 |
| Docker Image | 완성된 요리 (포장된 상태) |
| Container | 접시에 담아 실제로 먹는 것 |

레시피(Dockerfile)가 있으면 누구든, 어느 주방(환경)에서든 **똑같은 요리(이미지)** 를 만들 수 있습니다.

---

### Dockerfile의 역할 3가지

1. **기반 환경 지정** — 어떤 OS, 런타임 위에서 실행할지 정의 (예: Node.js 18, Python 3.11)
2. **앱 구성** — 필요한 파일 복사, 패키지 설치, 환경변수 설정
3. **실행 방법 정의** — 컨테이너가 시작될 때 어떤 명령을 실행할지 지정

---

## 2. Dockerfile 주요 명령어 (Instructions)

### 명령어 전체 흐름

```
FROM        ← 1. 베이스 이미지 선택 (재료 선택)
WORKDIR     ← 2. 작업 디렉토리 설정 (작업대 준비)
COPY / ADD  ← 3. 파일 복사 (재료 옮기기)
RUN         ← 4. 명령 실행 (조리)
EXPOSE      ← 5. 포트 열기 (창구 설정)
ENV         ← 6. 환경변수 설정 (조리 설정값)
CMD         ← 7. 컨테이너 시작 시 실행할 명령 (서빙)
```

---

### FROM — 베이스 이미지 지정

모든 Dockerfile은 반드시 `FROM`으로 시작합니다.  
**어떤 이미지를 기반으로 만들지** 지정합니다.

```dockerfile
FROM [image]:[tag]

FROM ubuntu:22.04         # Ubuntu 22.04를 베이스로 사용
FROM node:18-alpine       # Node.js 18 + Alpine Linux (경량 버전)
FROM python:3.11          # Python 3.11을 베이스로 사용
FROM scratch              # 아무것도 없는 빈 상태에서 시작 (고급)
```

> **alpine**이란?  
> 초경량 Linux 배포판입니다. 일반 ubuntu 대비 크기가 1/10 수준이라  
> Docker 이미지 크기를 줄이기 위해 자주 사용합니다.

---

### WORKDIR — 작업 디렉토리 설정

이후 명령어들이 실행될 **기본 경로**를 설정합니다.  
경로가 없으면 자동으로 생성합니다.

```dockerfile
WORKDIR [path]

WORKDIR /app              # /app 폴더를 작업 디렉토리로 설정
```

> `cd /app` 과 비슷하지만, 이후 모든 명령어에 영구적으로 적용됩니다.  
> 절대경로 사용을 권장합니다.

---

### COPY — 파일 복사

**로컬 파일이나 폴더**를 이미지 안으로 복사합니다.

```dockerfile
COPY [local-source] [container-destination]

COPY . .                          # 현재 폴더 전체를 WORKDIR로 복사
COPY package.json ./              # package.json만 복사
COPY src/ /app/src/               # src 폴더를 컨테이너의 /app/src/로 복사
```

> **패턴:** 의존성 파일을 먼저, 소스코드를 나중에 복사하는 것이 좋습니다.  
> 이렇게 하면 소스코드만 바뀔 때 의존성 설치 레이어를 재사용할 수 있어 빌드가 빠릅니다.

```dockerfile
# 좋은 예: 레이어 캐시 활용
COPY package*.json ./    # 의존성 파일 먼저
RUN npm install          # 의존성 설치 (소스코드 안 바뀌면 캐시 재사용)
COPY . .                 # 소스코드 나중에
```

---

### ADD — 파일 복사 (확장판)

`COPY`와 비슷하지만 두 가지 기능이 추가됩니다:
- URL에서 파일을 직접 다운로드 가능
- `.tar` 압축 파일을 자동으로 해제

```dockerfile
ADD [source] [destination]

ADD https://example.com/file.tar.gz /app/   # URL에서 다운로드
ADD archive.tar.gz /app/                    # tar 파일 자동 압축 해제
```

> **권장 사항:** 단순 파일 복사에는 `COPY`를 사용하세요.  
> `ADD`는 URL 다운로드나 tar 자동 해제가 필요한 경우에만 사용합니다.

---

### RUN — 명령 실행

이미지를 **빌드하는 시점**에 명령어를 실행합니다.  
주로 패키지 설치, 파일 생성 등에 사용합니다.

```dockerfile
RUN [command]

RUN npm install                         # npm 패키지 설치
RUN pip install -r requirements.txt     # Python 패키지 설치
RUN apt-get update && apt-get install -y curl   # apt 패키지 설치
```

> **`&&`로 연결하는 이유:**  
> `RUN` 명령어 하나가 이미지 레이어 하나를 만듭니다.  
> 여러 명령어를 `&&`로 연결하면 레이어 수를 줄여 이미지 크기가 작아집니다.

```dockerfile
# 비효율적: 레이어 3개 생성
RUN apt-get update
RUN apt-get install -y curl
RUN apt-get install -y git

# 효율적: 레이어 1개 생성
RUN apt-get update && apt-get install -y curl git
```

---

### EXPOSE — 포트 노출

컨테이너가 **사용할 포트를 문서화**합니다.  
실제로 포트를 여는 것은 `docker run -p`가 담당합니다.

```dockerfile
EXPOSE [port]

EXPOSE 3000     # Node.js 앱이 3000번 포트를 사용한다는 것을 명시
EXPOSE 8080
EXPOSE 80
```

> `EXPOSE`는 실제 네트워크 연결이 아닌 **"이 포트를 쓸 예정입니다"** 라는 선언입니다.  
> 실제로 외부에서 접근하려면 `docker run -p 3000:3000` 옵션이 필요합니다.

---

### ENV — 환경변수 설정

이미지 안에 **환경변수**를 설정합니다.  
빌드 시점과 컨테이너 실행 시점 모두에서 사용됩니다.

```dockerfile
ENV [key]=[value]

ENV NODE_ENV=production
ENV PORT=3000
ENV DB_HOST=localhost
```

> 민감한 정보(비밀번호, API 키)는 Dockerfile의 `ENV`에 직접 쓰지 마세요.  
> 대신 `docker run -e` 옵션이나 `.env` 파일로 런타임에 주입하는 것이 안전합니다.

---

### CMD — 컨테이너 시작 명령

컨테이너가 **시작될 때 기본으로 실행할 명령**을 지정합니다.  
Dockerfile에 하나만 작성할 수 있습니다 (여러 개면 마지막 것만 적용).

```dockerfile
CMD ["executable", "arg1", "arg2"]   # 권장 방식 (exec 형태)
CMD ["node", "app.js"]               # node app.js 실행
CMD ["python", "main.py"]            # python main.py 실행
CMD ["nginx", "-g", "daemon off;"]   # nginx 실행
```

> `docker run` 시 명령어를 추가하면 `CMD`는 무시됩니다.  
> 예: `docker run my-app node other.js` → CMD 대신 `node other.js` 실행

---

### ENTRYPOINT — 고정 실행 명령

`CMD`와 비슷하지만, `docker run`으로 명령어를 넘겨도 **덮어쓸 수 없습니다**.  
항상 실행되어야 하는 명령에 사용합니다.

```dockerfile
ENTRYPOINT ["executable", "arg"]

ENTRYPOINT ["node", "app.js"]
```

> **CMD vs ENTRYPOINT 차이:**
> | | docker run 시 명령어로 덮어쓰기 |
> |---|---|
> | `CMD` | 가능 (덮어씌워짐) |
> | `ENTRYPOINT` | 불가능 (항상 실행됨) |

## 3. 실전 Dockerfile 예시

### Node.js 앱

```dockerfile
# 1. Node.js 18 Alpine 이미지를 베이스로 사용
FROM node:18-alpine

# 2. 컨테이너 내부 작업 디렉토리를 /app으로 설정
WORKDIR /app

# 3. 의존성 파일을 먼저 복사 (캐시 최적화)
COPY package*.json ./

# 4. 의존성 설치
RUN npm install

# 5. 나머지 소스코드 복사
COPY . .

# 6. 앱이 3000번 포트를 사용한다고 명시
EXPOSE 3000

# 7. 컨테이너 시작 시 node app.js 실행
CMD ["node", "app.js"]
```

---

### Python 앱

```dockerfile
FROM python:3.11-slim

WORKDIR /app

# requirements.txt 먼저 복사 후 설치 (캐시 최적화)
COPY requirements.txt ./
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["python", "main.py"]
```

---

### Nginx 정적 사이트 서빙

```dockerfile
# 빌드 스테이지: Node.js로 React 앱 빌드
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# 서빙 스테이지: 빌드 결과물만 Nginx로 서빙
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

> 이처럼 빌드용 이미지와 실행용 이미지를 분리하는 것을 **멀티 스테이지 빌드**라고 합니다.  
> 최종 이미지에 빌드 도구가 포함되지 않아 이미지 크기가 크게 줄어듭니다.

## 4. .dockerignore — 복사 제외 파일 설정

`.gitignore`처럼, Docker 빌드 시 **이미지에 포함하지 않을 파일**을 지정합니다.

```
# .dockerignore 예시
node_modules/
.git/
.env
*.log
dist/
README.md
```

> `node_modules`를 제외하지 않으면 로컬의 수천 개 파일이 이미지로 복사되어  
> 빌드가 느려지고 이미지 크기가 커집니다.

## 5. 이미지 레이어(Layer) 이해

Dockerfile의 각 명령어는 **레이어(Layer)** 를 하나씩 만듭니다.

```
FROM node:18-alpine    ← Layer 1 (베이스)
WORKDIR /app           ← Layer 2
COPY package*.json ./  ← Layer 3
RUN npm install        ← Layer 4
COPY . .               ← Layer 5
CMD ["node", "app.js"] ← Layer 6 (메타데이터)
```

**레이어 캐시의 핵심 원칙:**  
변경된 레이어부터 아래 레이어는 모두 다시 빌드됩니다.  
→ **자주 바뀌는 내용(소스코드)은 아래쪽에, 잘 안 바뀌는 내용(의존성)은 위쪽에** 배치하세요.

```
자주 안 바뀜 → 위에 배치 (캐시 활용)
──────────────────────────────
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./     ← 패키지 목록: 가끔 바뀜
RUN npm install           ← 위가 안 바뀌면 캐시 재사용
──────────────────────────────
COPY . .                  ← 소스코드: 자주 바뀜
자주 바뀜 → 아래에 배치
```

## 6. 핵심 정리

| 명령어 | 실행 시점 | 역할 |
|---|---|---|
| `FROM` | 빌드 | 베이스 이미지 지정 |
| `WORKDIR` | 빌드 | 작업 디렉토리 설정 |
| `COPY` | 빌드 | 로컬 파일 → 이미지 복사 |
| `ADD` | 빌드 | COPY + URL/tar 지원 |
| `RUN` | 빌드 | 명령어 실행 (레이어 생성) |
| `EXPOSE` | 빌드 | 사용 포트 문서화 |
| `ENV` | 빌드/실행 | 환경변수 설정 |
| `CMD` | 실행 | 컨테이너 기본 시작 명령 (덮어쓰기 가능) |
| `ENTRYPOINT` | 실행 | 컨테이너 고정 시작 명령 (덮어쓰기 불가) |

> **핵심 한 문장:** Dockerfile은 "이 앱을 어떤 환경에서, 어떻게 준비하고, 어떻게 실행할지"를 코드로 기록한 설계도입니다.
