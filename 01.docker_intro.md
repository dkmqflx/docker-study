# Docker 입문 강의자료

---

## 1장. Docker란 무엇인가?

### Docker를 한 마디로 설명하면?

> **"애플리케이션을 컨테이너에 담아서 어디서든 똑같이 실행할 수 있게 해주는 기술"**

---

### 현실에서 만나는 문제

개발을 하다 보면 이런 상황을 자주 겪습니다:

- 내 컴퓨터에서는 잘 되는데, 팀원 컴퓨터에서는 안 된다
- 개발 환경에서는 되는데, 실제 서버에 올리면 오류가 난다
- 새 팀원이 와서 개발 환경 세팅하는 데 하루가 걸린다

**왜 이런 일이 생길까요?**

각자의 컴퓨터에 설치된 운영체제, 라이브러리, 프로그램 버전이 조금씩 다르기 때문입니다.

---

### 비유로 이해하기 — 선박 컨테이너

옛날에는 물건을 배에 실을 때 짐마다 크기와 모양이 달라서 싣고 내리는 데 시간이 엄청 걸렸습니다.

그런데 **표준 컨테이너**가 등장하면서 모든 게 바뀌었습니다:
- 짐을 컨테이너 안에 담으면 → 배든, 기차든, 트럭이든 **어디서나 똑같이 처리**
- 내용물이 뭐든 상관없이 **규격이 통일**되어 있음

**Docker 컨테이너도 똑같습니다:**
- 내 앱과 앱이 필요로 하는 모든 것(라이브러리, 설정, 코드)을 컨테이너에 담으면
- 내 노트북이든, 테스트 서버든, 클라우드든 **어디서나 똑같이 실행**

## 2장. Docker의 핵심 구성 요소

### 구성 요소 전체 그림

```
여러분이 작성       Docker가 만듦      Docker가 실행
──────────         ──────────         ──────────
Dockerfile    →   Docker Image   →   Container
(설계도)           (패키지/틀)         (실제로 돌아가는 것)
```

---

### 2-1. Dockerfile — "설계도"

앱을 어떻게 만들지 **단계별로 지시하는 스크립트**입니다.

**비유:** 요리 레시피와 같습니다.
- "재료는 Node.js 18 버전을 써라"
- "이 파일들을 이 폴더에 복사해라"
- "npm install 명령을 실행해라"

```dockerfile
# 예시 Dockerfile
FROM node:18          # 기본 재료: Node.js 18 이미지 사용
WORKDIR /app          # 작업 폴더 설정
COPY . .              # 내 코드를 컨테이너 안으로 복사
RUN npm install       # 의존성 설치
CMD ["node", "app.js"] # 앱 실행
```

---

### 2-2. Docker Image — "패키지/틀"

Dockerfile을 실행하면 Image가 만들어집니다.

**특징:**
- **읽기 전용** (수정 불가능)
- 앱 실행에 필요한 모든 것이 담긴 완성된 패키지
- 여러 컨테이너를 만들기 위한 **공통 틀**

**비유:** 붕어빵 틀과 같습니다.
- Image = 붕어빵 틀 (여러 번 사용 가능, 틀 자체는 변하지 않음)
- Container = 틀에서 찍어낸 붕어빵 (실제로 먹는 것)

---

### 2-3. Docker Container — "실제로 돌아가는 것"

Image로부터 생성된, **실제로 실행 중인 애플리케이션 인스턴스**입니다.

**특징:**
- Image를 기반으로 만들어지고, **읽기/쓰기 가능**
- 각 컨테이너는 서로 **격리**되어 있음 (다른 컨테이너 영향 안 받음)
- 필요하면 서로 **통신**할 수 있음

**비유:** 붕어빵 틀(Image)로 붕어빵(Container)을 여러 개 찍어낼 수 있습니다. 붕어빵끼리 서로 독립적이지만, 필요하면 함께 쟁반에 담아 제공(통신)할 수 있습니다.

---

### 2-4. Docker Engine — "실행 엔진"

Docker의 핵심으로, 컨테이너를 만들고 실행하고 관리하는 프로그램입니다.

3가지로 구성됩니다:

| 구성 요소 | 역할 | 비유 |
|---|---|---|
| **dockerd** (데몬) | 실제로 컨테이너를 실행하는 서버 | 주방장 |
| **REST API** | 데몬과 소통하는 인터페이스 | 주문서 |
| **CLI** | 사용자가 명령을 입력하는 도구 | 키오스크 |

---

### 2-5. Docker Hub — "앱 마켓스토어"

미리 만들어진 이미지를 올리고 내려받을 수 있는 **클라우드 저장소**입니다.

**비유:** 앱스토어(App Store)와 동일합니다.
- 남이 만들어둔 이미지를 **무료로 가져와서** 사용 가능
- 내가 만든 이미지를 **올려서 공유** 가능
- MySQL, Nginx, Redis 등 공식 이미지가 준비되어 있음

```bash
docker pull nginx   # Docker Hub에서 Nginx 이미지 다운로드
```

---

### 2-6. Docker Compose — "여러 컨테이너 한번에 관리"

실제 서비스는 보통 여러 컨테이너가 함께 동작합니다.

**예시:** 웹 서비스를 만든다면
- 웹 서버 컨테이너 (Nginx)
- 앱 서버 컨테이너 (Node.js)
- 데이터베이스 컨테이너 (MySQL)

이걸 하나하나 실행하면 복잡합니다. Docker Compose를 쓰면 **설정 파일 하나**로 한 번에 관리할 수 있습니다.

```yaml
# docker-compose.yml 예시
services:
  web:
    image: nginx
  app:
    build: .
  db:
    image: mysql
```

```bash
docker compose up   # 모든 컨테이너 한번에 실행
```

## 3장. VM(가상머신) vs Docker 컨테이너

Docker를 처음 배울 때 가장 많이 헷갈리는 부분입니다.

### 차이점 비교

| | **가상머신 (VM)** | **Docker 컨테이너** |
|---|---|---|
| 구조 | 운영체제 전체를 가상화 | 운영체제 커널을 공유 |
| 크기 | 수 GB | 수십 ~ 수백 MB |
| 시작 시간 | 수 분 | 수 초 (거의 즉시) |
| 성능 | 무거움 | 가벼움 |
| 격리 수준 | 완전 격리 | 프로세스 수준 격리 |

### 비유로 이해하기

**가상머신** = 집 안에 통째로 또 다른 집을 짓는 것 (방, 주방, 화장실 모두 따로)

**Docker** = 집 안에 방만 여러 개 만드는 것 (화장실, 주방은 공유하되 각 방은 독립적)

→ 방(컨테이너)이 훨씬 **빠르고 효율적**입니다.

## 4장. Docker 기본 명령어

### 필수 명령어 모음

```bash
# 이미지 관련
docker build -t my-app .        # 현재 폴더의 Dockerfile로 이미지 빌드
docker images                   # 로컬에 있는 이미지 목록 보기
docker pull nginx               # Docker Hub에서 이미지 다운로드
docker push my-app              # Docker Hub에 이미지 업로드

# 컨테이너 관련
docker run -d -p 8080:80 my-app # 컨테이너 실행 (백그라운드, 포트 연결)
docker ps                       # 실행 중인 컨테이너 목록
docker ps -a                    # 모든 컨테이너 목록 (중지된 것 포함)
docker stop <컨테이너ID>         # 컨테이너 중지
docker rm <컨테이너ID>           # 컨테이너 삭제
docker logs <컨테이너ID>         # 컨테이너 로그 보기
```

### `-p 8080:80` 이해하기

```
docker run -p 8080:80 nginx
              ↑     ↑
         내 PC 포트  컨테이너 포트

브라우저에서 localhost:8080으로 접속하면
컨테이너 내부의 80번 포트로 연결됩니다.
```

## 5장. Docker 실전 워크플로우

### 전체 흐름

```
1. Dockerfile 작성
        ↓
2. docker build → Image 생성
        ↓
3. docker run → Container 실행
        ↓
4. docker push → Docker Hub에 업로드
        ↓
5. 서버/클라우드에서 docker pull + docker run
```

### 간단한 실습 예시

Node.js 앱을 Docker로 실행해봅시다:

```dockerfile
# Dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "index.js"]
```

```bash
# 빌드하고 실행
docker build -t my-node-app .
docker run -d -p 3000:3000 my-node-app

# 브라우저에서 localhost:3000 접속하면 앱이 실행됨!
```

## 6장. Docker가 쓰이는 곳

| 분야 | 활용 방법 |
|---|---|
| **개발/테스트** | 팀 전체가 동일한 환경에서 개발 |
| **마이크로서비스** | 서비스별로 독립 컨테이너로 배포 |
| **CI/CD** | 자동화된 빌드·테스트·배포 파이프라인 |
| **멀티 클라우드** | AWS, GCP, Azure 어디서든 동일하게 실행 |
| **머신러닝** | GPU 환경과 라이브러리를 컨테이너로 공유 |

---

## 핵심 정리

| 개념 | 한 줄 요약 | 비유 |
|---|---|---|
| **Dockerfile** | 이미지를 만드는 설계도 | 요리 레시피 |
| **Image** | 실행 가능한 패키지/틀 | 붕어빵 틀 |
| **Container** | 실제로 실행 중인 앱 | 붕어빵 |
| **Docker Hub** | 이미지 저장소 | 앱스토어 |
| **Docker Compose** | 여러 컨테이너 통합 관리 | 오케스트라 지휘자 |

> **핵심 한 문장:** Docker는 "내 컴퓨터에서는 되는데..."를 영원히 없애기 위해 만들어진 기술입니다.
