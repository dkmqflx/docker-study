# Docker Volume 완전 정복

---

## 1. Docker Volume이란?

> **"컨테이너가 사라져도 데이터는 남아있게 해주는 기술"**

기본적으로 컨테이너 안에서 생성되는 모든 데이터는 컨테이너의 **쓰기 가능한 레이어(Writable Layer)** 에 저장됩니다.  
문제는 컨테이너를 삭제하면 이 레이어도 함께 사라진다는 것입니다.

```
컨테이너 실행 중    →    컨테이너 삭제
┌────────────────┐        ┌────────────────┐
│  컨테이너      │        │                │
│  ┌──────────┐  │  rm    │  (사라짐)      │
│  │ 데이터   │  │ ────▶  │                │
│  └──────────┘  │        │                │
└────────────────┘        └────────────────┘
    데이터 존재                데이터 소멸!
```

**Volume**은 데이터를 컨테이너 외부에 저장해서 이 문제를 해결합니다.

---

### 비유로 이해하기

컨테이너를 **호텔 방**이라고 생각해보세요.

- 호텔 방(컨테이너) 안에 물건을 두면, 체크아웃(컨테이너 삭제) 시 모두 사라집니다.
- 하지만 짐을 **호텔 창고(Volume)** 에 맡기면, 방을 바꾸거나 체크아웃해도 짐은 그대로입니다.
- 다음에 다시 체크인(새 컨테이너 실행)해도 창고에서 짐을 꺼내 사용할 수 있습니다.

---

### Volume이 필요한 대표적인 상황

| 상황 | 이유 |
|---|---|
| 데이터베이스 (MySQL, PostgreSQL) | 컨테이너 재시작 후에도 DB 데이터 유지 |
| 로그 파일 | 컨테이너 종료 후에도 로그 분석 가능 |
| 업로드 파일 | 사용자가 올린 파일이 삭제되면 안 됨 |
| 설정 파일 | 외부에서 설정을 주입하고 싶을 때 |

---

## 2. Volume의 핵심 특징 4가지

### 1. Persistence (영속성)
컨테이너가 중지되거나 삭제되어도 데이터가 그대로 유지됩니다.

### 2. Data Sharing (데이터 공유)
여러 컨테이너가 **같은 Volume을 동시에 마운트**해서 데이터를 공유할 수 있습니다.

```
컨테이너 A ─┐
             ├─▶ Volume (공유 데이터)
컨테이너 B ─┘
```

### 3. Host Independence (호스트 독립성)
Volume은 Docker가 관리하며, 특정 호스트 경로에 종속되지 않습니다.  
→ 어느 서버에서 실행해도 동일하게 동작합니다.

### 4. Ease of Management (간편한 관리)
`docker volume` 명령어로 생성·조회·삭제를 간단하게 할 수 있습니다.

---

## 3. Volume의 3가지 종류

Docker에서 데이터를 외부에 저장하는 방식은 크게 3가지입니다.

```
┌──────────────────────────────────────────────────┐
│                  컨테이너                         │
│              /data (마운트 포인트)                │
└──────────────────┬───────────────────────────────┘
                   │
        ┌──────────┼──────────┐
        ▼          ▼          ▼
  Anonymous     Named       Bind
   Volume       Volume      Mount
  (익명 볼륨)  (명명 볼륨)  (바인드 마운트)
  Docker 관리  Docker 관리  호스트 경로 직접 연결
```

---

### 3-1. Anonymous Volume (익명 볼륨)

이름 없이 **자동으로 생성되는 볼륨**입니다.  
Docker가 임의의 긴 ID로 이름을 붙입니다.

```bash
# -v 옵션에 컨테이너 경로만 지정하면 익명 볼륨이 생성됩니다.
docker run -v /data busybox

# Dockerfile에서 VOLUME 명령어로도 생성됩니다.
# VOLUME ["/data"]
```

**특징:**
- 이름이 없어 나중에 찾거나 재사용하기 어렵습니다.
- 컨테이너 삭제 시 함께 삭제될 수 있습니다.
- 임시 데이터나 빠른 테스트 용도로 사용합니다.

```bash
# 익명 볼륨은 이렇게 생긴 이름이 붙습니다.
# a1b2c3d4e5f6...  (무작위 해시값)
docker volume ls
```

> **권장 사항:** 익명 볼륨은 관리하기 어렵습니다. 가능하면 Named Volume을 사용하세요.

---

### 3-2. Named Volume (명명 볼륨)

**이름을 직접 지정**하는 볼륨입니다. 가장 권장되는 방식입니다.

```bash
# 볼륨을 먼저 생성합니다.
docker volume create my_volume

# 컨테이너 실행 시 볼륨을 연결합니다.
# -v [볼륨이름]:[컨테이너 경로]
docker run -v my_volume:/data busybox
```

**특징:**
- 이름으로 쉽게 찾고 재사용할 수 있습니다.
- 여러 컨테이너가 같은 볼륨을 공유할 수 있습니다.
- Docker가 내부적으로 관리하며 경로는 `/var/lib/docker/volumes/`에 저장됩니다.

```bash
# 실전 예시: MySQL 데이터 영속화
docker volume create mysql_data

docker run -d \
  --name mysql \
  -e MYSQL_ROOT_PASSWORD=secret \
  -v mysql_data:/var/lib/mysql \
  mysql:8.0

# 컨테이너를 삭제하고 다시 만들어도 DB 데이터는 mysql_data 볼륨에 그대로 남습니다.
docker rm -f mysql
docker run -d --name mysql -v mysql_data:/var/lib/mysql mysql:8.0
```

---

### 3-3. Bind Mount (바인드 마운트)

> **"호스트의 폴더와 컨테이너의 폴더를 하나로 연결(동기화)하는 것"**

연결된 순간부터 **양쪽은 완전히 같은 폴더**입니다. 한쪽에서 파일을 수정하면 다른 쪽에도 즉시 반영됩니다.

#### 비유 — 구글 드라이브

구글 드라이브를 내 PC에 설치하면, PC의 특정 폴더(`~/Google Drive`)와 클라우드가 실시간 동기화됩니다.

- PC에서 파일 수정 → 클라우드에도 즉시 반영
- 클라우드에서 파일 수정 → PC에도 즉시 반영

Bind Mount가 정확히 이 방식입니다.

```
내 PC의 폴더          컨테이너의 폴더
/Users/me/src   ←→   /app/src
(실제 파일)           (연결된 뷰)
```

#### 그림으로 보기

**Bind Mount 없을 때** — 컨테이너는 완전히 격리된 공간

```
내 PC                    컨테이너
┌─────────────┐          ┌─────────────┐
│ /Users/me/  │          │ /app/       │
│   app.js    │          │   app.js    │  ← 빌드 시 복사된 스냅샷
└─────────────┘          └─────────────┘
     ↑ 수정해도               변화 없음
```

**Bind Mount 있을 때** — 폴더가 하나로 연결됨

```
내 PC                    컨테이너
┌─────────────┐          ┌─────────────┐
│ /Users/me/  │          │ /app/       │
│   app.js ◀──┼──────────┼──▶ app.js  │  ← 같은 파일!
└─────────────┘          └─────────────┘
     ↑ 수정하면               즉시 반영!
```

#### 명령어

```bash
# -v [호스트 절대경로]:[컨테이너 경로]
docker run -v /host/path:/container/path busybox
#              ↑                  ↑
#         내 PC 경로       컨테이너 경로

# 현재 디렉토리를 컨테이너의 /app에 연결하는 예시
docker run -v $(pwd):/app node:18-alpine
```

#### 가장 흔한 사용 사례 — 개발 환경

Bind Mount 없이 개발하면 코드를 수정할 때마다 이미지를 다시 빌드해야 합니다.

```bash
# Bind Mount 없는 경우 — 소스 수정마다 빌드 필요
docker build -t my-app .   # 코드 바꿀 때마다 다시 빌드
docker run my-app
```

Bind Mount를 쓰면 로컬에서 저장만 하면 컨테이너 안에 즉시 반영됩니다.

```bash
# Bind Mount 있는 경우 — 저장하면 바로 반영 (핫 리로드)
docker run -d \
  -p 3000:3000 \
  -v $(pwd)/src:/app/src \
  node:18-alpine

# VS Code에서 파일 저장 → 컨테이너 안에 즉시 반영
```

**특징:**
- 호스트 경로에 직접 접근하므로 보안에 주의해야 합니다.
- Docker가 관리하지 않아 이식성이 낮습니다. (경로가 다른 서버에서는 경로가 다를 수 있음)

---

### 3가지 방식 비교

| | Anonymous Volume | Named Volume | Bind Mount |
|---|---|---|---|
| 이름 | 자동(해시) | 직접 지정 | 없음 (경로 사용) |
| 관리 주체 | Docker | Docker | 호스트 OS |
| 재사용 | 어려움 | 쉬움 | 가능 (경로로) |
| 데이터 위치 | Docker 내부 | Docker 내부 | 호스트 경로 |
| 주요 사용처 | 임시 데이터 | DB, 영속 데이터 | 개발 환경 |

## 4. Volume 관련 명령어

```bash
# 볼륨 생성
docker volume create my_volume

# 볼륨 목록 확인
# DRIVER, VOLUME NAME 정보를 보여줍니다.
docker volume ls

# 볼륨 상세 정보 확인
# 마운트 포인트(실제 저장 경로), 생성 시각, 라벨 등을 JSON으로 출력합니다.
docker volume inspect my_volume

# 볼륨 삭제
# 컨테이너가 사용 중인 볼륨은 삭제할 수 없습니다.
docker volume rm my_volume

# 사용하지 않는 볼륨 일괄 삭제 (디스크 정리 시 유용)
# 어떤 컨테이너에도 연결되지 않은 볼륨을 모두 삭제합니다.
docker volume prune
```

## 5. docker-compose에서 Volume 사용하기

실제 프로젝트에서는 `docker-compose.yml`에서 볼륨을 정의하는 경우가 많습니다.

```yaml
services:
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: secret
    volumes:
      # Named Volume: mysql_data 볼륨을 컨테이너의 /var/lib/mysql에 연결
      - mysql_data:/var/lib/mysql

  app:
    build: .
    volumes:
      # Bind Mount: 로컬 소스코드를 컨테이너의 /app에 연결 (개발용)
      - ./src:/app/src

# 사용할 Named Volume을 최하단에 선언합니다.
volumes:
  mysql_data:
```

## 6. 베스트 프랙티스 (Best Practices)

### DO (권장)

```bash
# 데이터 영속화에는 Named Volume을 사용하세요.
docker volume create db_data
docker run -v db_data:/var/lib/mysql mysql

# 개발 시 소스코드 동기화에는 Bind Mount를 활용하세요.
docker run -v $(pwd):/app my-app

# 불필요한 볼륨은 정기적으로 정리해서 디스크를 관리하세요.
docker volume prune
```

### DON'T (주의)

```bash
# 민감한 데이터가 있는 호스트 경로를 Bind Mount로 무분별하게 연결하지 마세요.
# 잘못하면 컨테이너에서 호스트 전체를 접근할 수 있습니다.
docker run -v /:/host my-app  # 위험!

# Anonymous Volume에 중요한 데이터를 저장하지 마세요.
# 나중에 어느 볼륨인지 찾기 어렵습니다.
docker run -v /data my-app    # 중요한 데이터라면 Named Volume을 쓰세요.
```

## 7. 핵심 정리

| 개념 | 핵심 |
|---|---|
| **Volume의 목적** | 컨테이너 삭제 후에도 데이터 유지 |
| **Anonymous Volume** | 이름 없이 자동 생성, 임시 용도 |
| **Named Volume** | 이름 지정, 재사용 가능, **가장 권장** |
| **Bind Mount** | 호스트 경로 직접 연결, 개발 환경에 적합 |

> **핵심 한 문장:** Volume은 컨테이너의 짧은 생명주기와 데이터의 영속성 사이의 간극을 메워주는 핵심 기능입니다.
